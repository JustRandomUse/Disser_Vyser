# Введение

Для удобства и гибкости настройки введено понятие «набора данных». Набор данных описывает подмножество данных с площадок наблюдения, доступное пользователю. Структура:

- Список площадок наблюдения.
- Список доступных показателей.
- Ограничения на доступный период данных.
- Скользящее ограничение на глубину доступного архива в зависимости от детализации данных («сырые»/час/день). Например, сырые данные могут быть доступны не старее 1 недели от текущей даты.

Права доступа к наборам данных задаются для отдельных пользователей.

Названия видов данных, которые упоминаются в документе:

- Актуальные данные – данные, которые пришли последними.
- «Сырые» данные – данные, которые приходят с датчиков без агрегации, дата-время не привязана к временной сетке.
- Агрегированные данных – данные, вычисленные на основе сырых данных, и уложенные на строгую временную сетку (доступны интервалы: час, день, месяц).

# Сервис

Веб-сервис реализован на основе REST подхода. Запрос передается HTTP методами GET/POST/DELETE с параметрами, результат формируется в формате JSON.

Базовый URL: http://sensor.krasn.ru/hub/api/3.0/.

У всех запросов есть следующие общие параметры.

| uid | Ключ пользователя | Можно передавать GET параметром или в заголовке запроса (лучше). Параметр обязательный для всех запросов. |
| --- | --- | --- |
| opt-date | Формат даты-времени | Поддерживаются значения:<br>odbc – Формат YYYY-MM-DD HH:mm:ss (например, 2022-01-01 12:20:00). Используется по-умолчанию.<br>ts – Unix timestamp (например, 1640970000). |

Формат ответа:

```json
{
  "status": {
    "code": 1,
    "message": "OK"
  },
  "data": <data>
}
```

Поле code в статусе ответа - число, если code<0 – ошибка. Содержимое поля data зависит от запроса.

## Наборы данных

### Получить список наборов данных

```http
GET /sets
```

Поля ответа:

- id и code являются уникальными и могут использоваться для идентификации. Лучше использовать code, это даст дополнительную гибкость.
- name – наименование набора данных.
- short_name – краткое наименование.
- description – описание набора данных.
- last_time – дата время самого последнего значения показателя.

Пример ответа:

```json
{
  "status": {
    "code": 1,
    "message": "OK"
  },
  "data": [
    {
      "id": 2,
      "code": "min",
      "name": "Министерство",
      "last_time": "2022-08-18 23:00:00",
      "short_name": "M"
    },
    {
      "id": 1,
      "code": "air",
      "name": "Сеть КНЦ",
      "last_time": "2022-08-18 23:16:48"
    },
    {
      "id": 10,
      "code": "hpp-mtp5",

      "description": "Данные с температурных профилемеров MTP-5",
      "last_time": "2022-08-18 23:15:00"
    }
  ]
}
```

### Получить описание набора данных

```http
GET /sets/<id|code>
```

Параметры:

- <id|code> - идентификатор или код набора данных

Поля:

- code – кодовое название набор данных
- name – наименование набора данных
- short_name – краткое наименование
- description – описание набора данных
- last_time – дата-время последних данных
- period_begin – начало периода, за который доступны данные
- period_end – конец периода доступных данных, может быть null
- period_limit – скользящее ограничение на доступный архив в зависимости от детализации данных. Указывается в днях, отсчитывается от текущей даты.
  - raw – «сырые» данные
  - hour – часовые данные
  - day – дневные данные
- sites – список доступных площадок
  - id – идентификатор площадки
  - name – наименование
  - timing – режим сбора данных
    - regular – данные поступают регулярно
    - variable – измерения выполняются с перерывами
  - geom_x – долгота
  - geom_y – широта
  - geom_alt – высота над уровнем моря (может быть null), в метрах
  - geom_dir – направление установки в градусах относительно севера [0,360)
- indicators – список доступных показателей
  - id – уникальный идентификатор
  - code – кодовый идентификатор
  - name – наименование
  - units – единицы измерения
  - tag – произвольное целое число, может использоваться для удобства обработки однотипной серии показателей (например, распределения температуры по высоте)
- config – произвольная json структура с настройками, задается администратором

Содержимое полей code, name, units, tag у показателей может быть задано индивидуально для каждого набора данных. Порядок сортировки тоже.

### Получить актуальные данные

```http
GET /sets/<id|code>/data/last?<parameters>
```

Значения показателей с площадки (site) могут изменяться не синхронно. Например, метеоданные обновляются каждые 20 минут, а концентрация примесей – каждый час. В поле «time» выдается последнее время, когда пришли какие-либо данные по площадке (максимальное по всем показателям). Если значение показателя обновлялось слишком давно (пока установлен порог – 1 сутки), то он не попадает в выборку. Это сделано для корректной обработки ситуации, когда один из датчиков выходит из строя и перестает выдавать данные. Получить полные данные с фиксацией индивидуального времени получения и учетом устаревших можно через альтернативный запрос (см. след. раздел).

Параметры (фильтры):

| sites | Площадки | Выдавать данные только с указанных площадок: <br>«site.id, site.id, …». По-умолчанию – все. | Например, «3837, 3841» |
| --- | --- | --- | --- |
| indicators | Показатели | Выдавать только указанные показатели: «indicator.code, indicator.code, …». По-умолчанию – все. | Например, «m-t, p-pm2» |

Пример ответа:

```json
{
    "status": {
        "code": 1,
        "message": "OK"
    },
    "data": [
        {
            "site": 3837,
            "time": "2022-08-26 12:56:16",
            "i355": 21,
            "i371": 0.13,
            "m-h": 99,
            "m-p": 735.4,
            "m-t": 15.2,
            "p-pm10": 7,
            "p-pm2": 7

        {
            "site": 3841,
            "time": "2022-08-26 12:59:10",
            "i355": 34,

            "m-p": 741.5,
            "m-t": 15.8,
            "p-pm10": 11,
            "p-pm2": 10
        }
    ]
}
```

### Получить объем данных (агрегированных)

```http
GET /sets/<id|code>/data/stat-ext?<parameters>
```

По каждому показателю на каждой площадке выдается:

- avg – среднее значение;
- min – минимальное значение;
- max – максимальное значение;
- std – среднеквадратичное отклонение;
- cnt – количество измерений.

Запрос выполняется по таблице с агрегированными данными. Например, если будет задан интервал в 1 час, то запрос выдаст количество (поле cnt) часов за период, в которых есть данные.

## Проекты

### Получить список проектов

```http
GET /projects
```

Поля ответа:

- id и code являются уникальными и могут использоваться для идентификации. Лучше использовать code, это даст дополнительную гибкость.
- name – наименование проекта.
- short_name – краткое наименование.
- description – описание.
- last_time – дата время самого последнего значения показателя.

### Получить описание проекта

```http
GET /projects/<id|code>
```

Параметры:

- <id|code> - идентификатор проекта

Поля:

- code – кодовое название набор данных
- name – наименование набора данных
- short_name – краткое наименование
- description – описание набора данных
- last_time – дата-время последних данных
- period_begin – начало периода, за который доступны данные
- period_end – конец периода доступных данных, может быть null
- sites – список доступных площадок
  - id – идентификатор площадки
  - name – наименование
  - geom_x – долгота
  - geom_y – широта
  - geom_alt – высота над уровнем моря (может быть null), в метрах
- indicators – список доступных показателей
  - id – уникальный идентификатор
  - code – кодовый идентификатор
  - name – наименование
  - units – единицы измерения
  - tag – произвольное целое число, может использоваться для удобства обработки однотипной серии показателей (например, распределения температуры по высоте)

Содержимое полей code, name, units, tag у показателей может быть задано индивидуально для каждого набора данных. Порядок сортировки тоже.

### Отправить данные

```http
POST /projects/<id|code>/data .
```

Параметры:

- <id|code> - идентификатор проекта

В теле запроса передаются данные в JSON формате в виде массива записей. Каждая запись содержит поля:

- time – дата-время данных
- site – идентификатор площадки (обязательно из указанного проекта)
- остальные элементы трактуются как пары «код-показателя»: «значение показателя».

